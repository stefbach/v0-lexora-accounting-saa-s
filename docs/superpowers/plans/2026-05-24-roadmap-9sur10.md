# Plan d'action — Lexora 6.9/10 → 9/10

**Goal:** Faire passer l'application Lexora de 6.9/10 (audit du 2026-05-24) à 9/10 production-ready.

**Architecture:** Orchestration à 3 niveaux : chef d'orchestre (Claude principal) → 50 agents spécialisés répartis en 5 vagues thématiques → chaque agent peut spawner jusqu'à 5 sous-agents pour parallélisation interne.

**Tech Stack:** Next.js 16 App Router, TypeScript 5.7, Supabase (`dqepdoimpqhmuhkklxva`), Tailwind v4, Radix UI/shadcn, Vercel, Playwright (E2E à venir), Vitest.

---

## 🔗 Dépendances entre vagues

```
V1 (Sécurité)  ─┐
                ├─→ V5 (Tests E2E + validation)
V2 (Fonctionnel)┤
                │
V3 (Code Qty) ──┼─→ V4 (UX/UI) ─→ V5
                │
V4 démarre dès que V1 RLS Phase 2 OK (pour ne pas casser le multi-tenant)
```

**Règle anti-conflits** : chaque agent reçoit une liste explicite de fichiers EXCLUSIFS (pas de chevauchement). Sous-agents partagent le périmètre du parent.

## ⚠️ Risques & garde-fous

1. **Conflits de fichiers** : chaque agent reçoit liste EXCLUSIVE de fichiers
2. **Migrations Supabase** : règle CLAUDE.md = pas d'`apply_migration` sans confirmation. Les agents ÉCRIVENT les SQL, l'application reste manuelle
3. **Build doit rester vert** : checkpoint après chaque vague (build + tests)
4. **Vagues séquentielles** : V1→V2→V3→V4→V5 (pas tout en parallèle, dépendances respectées)
5. **PR séparée par vague** : 5 PRs au final (V1 sécu en priorité hotfix-mergeable, puis le reste)
6. **Rollback possible** : chaque commit par agent = atomique, revert facile
